Introduction

Micro Services Architectural style is popular now-a-days just like the AGILE Methodology but both are being misused. Consuming a well designed Micro-Service is easy but when you get to work on consuming some badly designed Micro-Service then things start getting interesting. Micro-Service and RESTful web services were here to solve certain problems but with every solution a new problem creeps in. If we ignore this problem soon you Micro-Service will have all the Design Issues due to the ripple effect of the downstream services. Lets explore this topic today with some examples.

Consider that you have to Desing a new Micro-Service and you are asked to consume a downstream Micro-Service which is poorly Designed. Obviously with Design I mean API resources and their behaviour. A poor design if limited to the backend system is ok but if API resources expose that bad desing then its impacting your Design too. Lets take an example that you are designing an E-Commerce platform and there is an Order Management Micro-Service which you need to consume for your requirement. Suppose that the APIs desinged by that system are like below:

  • CREATE Order

    POST/orders

  • UPDATE Order

    PUT/orders/{OrderId}

  • GET Order Details

    GET/orders/{OrderId}

  • GET All Orders

    GET/orders/{OrderId}
