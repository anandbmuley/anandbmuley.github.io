Introduction

A final class denotes that Design is finalized and cannot be extended further. Having final classes for a finalized design will benefit in clearly communicating other developers but will it impact the extensibility principle ? What are the cases which makes a class a potential candidate for finalization ? Are we not adhering to the Design principles if we make the classes final ? What is need to clear communication among Developers ?

Now lets come back to the topic. A final class has a final keyword which can show that its not be extended or moreover it cannot be extended. Its mostly used for the cases where a library owner does not want the consumer of the library to extend it an create it extensions which may lead to un-predictable behaviour. What are some of the examples of these kind of classes ? I will give a real life example for this then we will try to find such examples with some correlation.

The final keyword restricts extension or inheritence. In real life we know that we cannot have any child class of Color class. This is applicable if the inheritence criteria is based on the wavelengths of light. Red colored light has a wavelength of ~ 700–635 nm and Yellow has ~ 635–590 nm (where nm - nanometers). In case of the libraries lets say you have created a parser which can parse XML, JSON and CSV files. You have the library designed for supporting these file formats and dedicated classes are there for them. You don't want the consumer of the library to extend these classes as it may create some undesired results. Hence you will make them final. Simply put, you will make all those classes final which you don't want the consumer of the library to extend. You will let those classes non-final which support extension. In a way you are in control of the extensions supported by your library.

A final class can also be created in the same project when we realize the fact that it should not be extended further. Possible scenarios are :

  • Object Modelling for the Leaf Child in the Parent Child hierarchy - It is the case of modelling hierarchy of Domain Objects where in we know in the hierarchy for the Domain model it should not be extended further
  • Shallow hierarchy for Strategy Pattern - A strategy pattern is behavioural design pattern used to dynamically invoke a particular algorithm. Lets not get confused with algorithm here. Its a behavioural design pattern which has the highest level of abstraction of that of the common behaviour. Hence it can be used for selecting which mapper to be invoked based on the need. But in these cases when we increase the depth of specialization then things are unmanageable. Hence when we want the depth of hierarchy to be controlled then we should go for a final class

Summary

A final class is an indication that its the least child in the hierarchy and its design is final and could not be extended. Hence we have to be cautious while creating final classes but we should not be hesitant to make a class final.
